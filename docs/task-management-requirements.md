# タスク管理機能 要件定義

## 1. 背景・目的

GAS Pomodoro はポモドーロタイマー + 作業記録のツールだが、現状「何のために作業したか」をプロジェクト・タスク単位で追跡する仕組みがない。カテゴリ（作業/学習/その他）は作業の種類を分類するが、具体的なプロジェクトやタスクへの時間配分は把握できない。

タスク管理機能を追加し、ポモドーロ記録をタスクに紐づけることで、プロジェクト・案件・タスクごとの作業時間を可視化する。

## 2. 要件概要

### 2.1 階層構造

3階層のタスク管理構造を導入する。

```
プロジェクト
├── 案件A
│   ├── タスク1
│   └── タスク2
├── 案件B
│   └── タスク3
└── タスク4（プロジェクト直属）
```

- **プロジェクト**: 最上位の分類単位。色を持つ。
- **案件**: プロジェクト配下の中間グループ。
- **タスク**: 最小の作業単位。6つのステータス（ToDo / Doing / Review / Done / Pending / Docs）を持つ。
- タスクはプロジェクトに直属することもできる（案件を経由しない）。

### 2.2 ポモドーロ記録との紐づけ

- ポモドーロ完了時の記録フォームでタスクを選択できる（任意）。
- 既存のカテゴリ選択とは独立して併存する。
- タスクごとの合計作業時間を集計・表示できる。

### 2.3 wiki（リッチテキストドキュメント）

- プロジェクト・案件・タスクの各ノードに、リッチテキスト（TipTap エディタ）のドキュメントを持つ。
- メモ機能と同じエディタを使用する。

### 2.4 タスク状態管理

- タスクは以下の6ステータスを持つ（各ステータスに固定色を定義）:

  | ステータス | 表示名 | 色 | 用途 |
  |-----------|--------|-----|------|
  | ToDo | 未着手 | `#9e9e9e`（グレー） | まだ手をつけていないタスク |
  | Doing | 作業中 | `#e53935`（レッド） | 現在取り組み中。既存の `--work-color` と統一 |
  | Review | レビュー | `#fb8c00`（オレンジ） | レビュー待ち・確認待ち |
  | Done | 完了 | `#43a047`（グリーン） | 完了済み。既存の `--break-color` と統一 |
  | Pending | 保留 | `#7e57c2`（パープル） | 一時的にブロック・保留中 |
  | Docs | ドキュメント | `#1e88e5`（ブルー） | ドキュメント（後述） |

  - 色はシステム固定（ユーザーによるカスタマイズ不可）。
  - テーブルやツリーでの表示時に、カテゴリバッジと同様に色付きラベルとして表示する。
- プロジェクト・案件は状態を持たず、`isActive` によるアーカイブのみ。
- ステータスの遷移に制約はない（任意のステータスから任意のステータスへ変更可能）。

### 2.5 Docs ステータス（ドキュメントとしてのタスク）

- `status=Docs` のタスクは、通常のタスクではなくプロジェクト・案件に紐づく**ドキュメント**として扱う。
- 設計書・議事録・手順書などを、タスクの仕組み（階層紐づけ・wiki エディタ・ポモドーロ紐づけ）を利用して管理する。
- プロジェクト・案件の `content`（概要 wiki）とは別に、複数の Docs タスクを作成できる。
- Docs タスクにもポモドーロを紐づけ可能（ドキュメント作成に費やした時間を計測できる）。
- タスク一覧では通常タスクと Docs タスクを視覚的に区別して表示する。

### 2.6 メンション（クロスリファレンス）

wiki エディタ内で他のエンティティをメンションし、クリックでナビゲーションできる。既存のメモ間メンション（`@`）を拡張し、タスク・プロジェクトへのメンションを追加する。

#### トリガー文字と対象

| トリガー | 対象 | 表示例 | 候補リスト |
|----------|------|--------|-----------|
| `@` | メモ | `@議事録0215` | 既存のメモ一覧（変更なし） |
| `#` | タスク | `#API設計` | 全プロジェクトの未アーカイブタスク。プロジェクト名を併記して同名タスクを区別 |
| `!` | プロジェクト / 案件 | `!クライアントA` `!クライアントA::フェーズ2` | プロジェクト一覧 + 案件一覧 |

#### 候補リストの表示

- **`#` タスクメンション**: 入力文字でフィルタ。各候補に `プロジェクト名 > 案件名 > タスク名` の階層を薄い文字で併記し、同名タスクを区別する。
- **`!` プロジェクト/案件メンション**: プロジェクトと案件をフラットに一覧表示（プロジェクトは色インジケーター付き、案件は親プロジェクト名を併記）。`!PJ::案件` の階層記法は将来対応。

#### クリック時のナビゲーション

- `#タスク名` クリック → タスクタブに切り替え、ツリーで該当タスクを選択・表示。
- `!プロジェクト名` クリック → タスクタブに切り替え、ツリーで該当プロジェクトを選択・表示。
- `!PJ::案件名` クリック → タスクタブに切り替え、ツリーで該当案件を選択・表示。
- `@メモ名` クリック → メモタブに切り替え（既存動作、変更なし）。
- ナビゲーション履歴は既存の `navStack` を拡張して戻るボタンに対応。

#### 使用可能な場所

全エディタ（メモ・タスク wiki・プロジェクト wiki・案件 wiki・作業記録の description）で `@` `#` `!` すべてのメンションが使用可能。

#### ストレージ

- Markdown 内に `[@メモ名](mention://memo-uuid)` `[#タスク名](task://task-uuid)` `[!PJ名](project://project-uuid)` `[!PJ::案件](case://case-uuid)` の形式で保存。
- UUID ベースなので、名前変更時もリンクが切れない。

### 2.7 開始日時・期限

- タスクに **開始日時（startedAt）** と **期限（dueDate）** を設定できる。
- どちらも任意（空でも可）。
- **startedAt の設定方法:**
  1. **手動設定**: タスク詳細画面から任意の日時を入力。
  2. **自動設定**: ポモドーロ記録の保存時に taskId が指定され、かつそのタスクの startedAt が空の場合、該当ポモドーロの startTime を自動でセットする。
  - 手動で既に設定済みの場合は自動設定で上書きしない。
- **dueDate**: 日付のみ（時刻なし）。タスク詳細画面から手動設定。

## 3. データモデル

### 3.1 新規シート

#### Projects シート（8列）

| 列 | カラム名 | 型 | 説明 |
|----|----------|----|------|
| A | id | UUID | 一意識別子 |
| B | name | String | プロジェクト名 |
| C | content | String | wiki ドキュメント（Markdown） |
| D | color | String | 表示色（#hex） |
| E | sortOrder | Number | 表示順 |
| F | isActive | Boolean | false=アーカイブ |
| G | createdAt | String | 作成日時（ISO 8601） |
| H | updatedAt | String | 更新日時（ISO 8601） |

#### Cases シート（8列）

| 列 | カラム名 | 型 | 説明 |
|----|----------|----|------|
| A | id | UUID | 一意識別子 |
| B | projectId | UUID | 親プロジェクトID（FK） |
| C | name | String | 案件名 |
| D | content | String | wiki ドキュメント（Markdown） |
| E | sortOrder | Number | プロジェクト内の表示順 |
| F | isActive | Boolean | false=アーカイブ |
| G | createdAt | String | 作成日時 |
| H | updatedAt | String | 更新日時 |

#### Tasks シート（14列）

| 列 | カラム名 | 型 | 説明 |
|----|----------|----|------|
| A | id | UUID | 一意識別子 |
| B | projectId | UUID | 所属プロジェクトID（FK） |
| C | caseId | UUID or "" | 所属案件ID（空文字=プロジェクト直属） |
| D | name | String | タスク名 |
| E | content | String | wiki ドキュメント（Markdown） |
| F | status | String | "todo" / "doing" / "review" / "done" / "pending" / "docs" |
| G | sortOrder | Number | 案件内/直属グループ内の表示順 |
| H | isActive | Boolean | false=削除 |
| I | createdAt | String | 作成日時 |
| J | completedAt | String | 完了日時（Done 以外に変更時はクリア、空文字=未完了） |
| K | startedAt | String | 開始日時（手動設定 or 初回ポモドーロ紐づけ時に自動設定、空文字=未開始） |
| L | dueDate | String | 期限日（YYYY-MM-DD、空文字=期限なし） |
| M | updatedAt | String | 更新日時（ISO 8601） |

### 3.2 既存シートの拡張

#### PomodoroLog（+1列）

| 列 | カラム名 | 型 | 説明 |
|----|----------|----|------|
| P | taskId | UUID or "" | 紐づくタスクID（空文字=紐づけなし） |

- 既存の15列（A〜O）は変更なし。後方互換性あり。

### 3.3 時間集計ロジック

```
タスク時間 = SUM(PomodoroLog.actualDurationSeconds)
             WHERE taskId = task.id AND type = 'work'

案件時間   = SUM(配下タスクの時間)
プロジェクト時間 = SUM(直属タスク + 全案件配下タスクの時間)
```

- PomodoroLog には taskId のみ保存し、projectId/caseId は Tasks シートから逆引きする（非正規化を避ける）。

## 4. UI/UX 設計

### 4.1 全体構成

右サイドバーのタブバーに「タスク」タブを追加する（計6タブ）。

```
[メモ] [タスク] [作業記録] [中断] [記録詳細] [設定]
        ↑ 新規追加                 ↑ 旧「ビュー」から改名
```

- タスクタブはメモタブの直後に配置。
- タスクタブは全フェーズで常時表示（メモタブと同様）。
- 旧「ビュー」タブは「記録詳細」に改名（過去のポモドーロ記録の閲覧・分析画面）。

### 4.2 タスクタブ内のレイアウト — 2カラム構成

タスクタブは左右2カラムで構成する（メモタブの2カラムレイアウトと同方式）。モバイル対応は不要。

```
┌─────────────────┬──────────────────────────────────┐
│  ツリーサイド     │  コンテンツエリア                  │
│  バー            │                                   │
│                  │  選択したノードの詳細を表示         │
│  🟦 プロジェクトA │                                   │
│  ├── 案件X       │                                   │
│  │   ├ タスク1   │                                   │
│  │   └ タスク2   │                                   │
│  ├── 案件Y       │                                   │
│  │   └ タスク3   │                                   │
│  └── タスク4     │                                   │
│  🟩 プロジェクトB │                                   │
│  └── ...         │                                   │
│                  │                                   │
│ [+ プロジェクト]  │                                   │
└─────────────────┴──────────────────────────────────┘
```

#### ツリーサイドバー（左）

- プロジェクト > 案件 > タスクの3階層ツリーを折りたたみ表示。
- プロジェクト名の左にカラーインジケーター（プロジェクトの色）。
- ノードクリックで右側のコンテンツエリアに詳細を表示。
- 各階層に「+」ボタンで子ノードを追加。
- アーカイブ済みノードはデフォルト非表示。
- Docs ステータスのタスクは通常タスクと視覚的に区別（アイコンなど）。
- **削除・アーカイブ操作**: ノード右クリック（またはロングタップ）でコンテキストメニューを表示し、「アーカイブ」「名前変更」等の操作を提供（メモのコンテキストメニューと同様の方式）。
- **並び替え**: ドラッグ＆ドロップでツリー内のノードを並び替え可能。同一階層内での移動のみ（例: 同じプロジェクト内の案件同士、同じ案件内のタスク同士）。sortOrder は移動後にドロップ位置に基づいて再計算（0-indexed 連番を再割り当て）。メモの並び替えと同方式（`pendingOrder` + `flushOrderSave` パターン）で実装。
- **タスクの案件間・プロジェクト間移動**: 初期リリースでは対応しない（将来対応）。実装コストが高く必須ではないため。移動が必要な場合は、タスクを新規作成し元のタスクをアーカイブする運用で代替。
- **初期選択状態**: 前回選択したノード ID を localStorage に保存し、次回起動時に復元する（メモタブと同方式）。該当ノードが存在しない場合（アーカイブ済み等）は未選択状態とし、コンテンツエリアに「プロジェクトまたはタスクを選択してください」のプレースホルダーを表示する。
- **スクロール位置キャッシュ**: メモタブの `memoScrollCache` と同じパターンで、ノードごとにwikiエディタのスクロール位置をメモリ上にキャッシュする。ツリーでノードを切り替える際に現在のスクロール位置を保存し、再度そのノードを選択したときに復元する（`saveMemoScroll` / `restoreMemoScroll` と同方式）。タスクテーブルのスクロール位置も同様にキャッシュする。

  - 各記録の2行目に description の1行目をそのまま表示（Markdownのstrip不要）。
  - 記録行クリックで記録詳細タブに遷移し、該当レコードを表示する。

### 4.4 記録フォームへのタスクピッカー追加

ポモドーロ完了時の記録フォーム（作業記録タブ）にタスク選択を追加:

```
作業記録タブ
├── [カテゴリ ▼] [前回をコピー]    ← 既存
├── [タスク ▼]                     ← 新規追加
├── エディタ                       ← 既存
└── アクションボタン               ← 既存
```

- タスクピッカーは階層構造でタスクを表示:
  ```
  プロジェクトA
    案件X
      タスク1
      タスク2
    タスク3（直属）
  プロジェクトB
    ...
  ```
- Done・Docs 以外のタスクのみ表示（完了タスクとドキュメントは除外）。
- 選択は任意（タスクなしでも記録可能）。
- 既存のカテゴリピッカーとは独立。
- **データソース**: IndexedDB の `tasks` / `projects` / `cases` ストアから直接取得。タスクタブの表示状態に依存しない。アプリ起動時の `getAllTaskData()` で IndexedDB にデータが格納されるため、タスクタブ未表示でも動作する。

### 5.2 主要 API

#### 一括取得
- `getAllTaskData(): { projects: ProjectMetadata[], cases: CaseMetadata[], tasks: TaskMetadata[] }` — アクティブな全プロジェクト・案件・タスクのメタデータを一括取得（content 除く）。初回ロードおよびタブ起動時の同期で使用。1回の API 呼び出しで完結し、N+1 問題を回避する。

#### プロジェクト
- `getProjects(): ProjectMetadata[]` — アクティブなプロジェクト一覧（content除く）。個別取得が必要な場合のみ使用。
- `getProjectContent(id): { id, content, updatedAt }` — プロジェクトの wiki 内容
- `addProject(name, color): Project` — プロジェクト追加
- `updateProject(id, fields)` — 名前・色・content の更新
- `archiveProject(id)` — isActive=false に設定

#### 案件
- `getCases(projectId): CaseMetadata[]` — プロジェクト配下の案件一覧
- `getCaseContent(id): { id, content, updatedAt }` — 案件の wiki 内容
- `addCase(projectId, name): Case` — 案件追加
- `updateCase(id, fields)` — 名前・content の更新
- `archiveCase(id)` — isActive=false に設定

#### タスク
- `getTasks(projectId): TaskMetadata[]` — プロジェクト配下の全タスク（案件配下含む、content 除く）
- `getTaskContent(id): { id, content, updatedAt }` — タスクの wiki 内容
- `addTask(projectId, caseId, name): Task` — タスク追加
- `updateTask(id, fields)` — 名前・content・status・startedAt・dueDate 等の更新。status が含まれる場合、Done 設定時に `completedAt` を自動記録、Done 以外への変更時に `completedAt` をクリア。
- `deleteTask(id)` — isActive=false に設定

#### 並び替え
- `reorderProjects(orderedIds)` — プロジェクトの sortOrder を一括更新
- `reorderCases(projectId, orderedIds)` — プロジェクト内の案件の sortOrder を一括更新
- `reorderTasks(parentId, orderedIds)` — 案件内（または直属グループ内）のタスクの sortOrder を一括更新

並び替えは ID 配列をサーバーに送信し、サーバー側で各行の sortOrder を書き換える（メモの `updateMemoSortOrders` と同方式）。IndexedDB 側は即時反映し、サーバー同期は `pendingOrder` + `flushOrderSave` パターンで非同期処理する。並び替えは複数レコードの一括更新のため、個別レコードの `_dirty` フラグではなく専用の同期フローで管理する。

#### 時間集計
- `getTaskTimeReport(projectId?): TimeReport` — taskId で PomodoroLog を集計し、タスク/案件/プロジェクトごとの合計時間を返す

#### ポモドーロ記録
- `getTaskPomodoroRecords(taskId): PomodoroRecord[]` — タスクに紐づく作業記録一覧

### 5.3 startedAt 自動設定

- **実行箇所**: サーバーサイド（`saveRecord()` 内）。
- ポモドーロ記録の保存時に `taskId` が指定されている場合、Tasks シートから該当タスクの `startedAt` を確認し、空であれば当該ポモドーロの `startTime` をセットする。
- サーバー側で一貫して処理することで、IndexedDB 同期と独立した整合性を保証する。

### 5.4 InitService.ts 変更

- `ensureSheets()` に Projects / Cases / Tasks シートの作成ロジックを追加。
- PomodoroLog シートのヘッダーに列 P（taskId）を追加。

## 6. 既存機能との関係

| 既存機能 | 影響 |
|----------|------|
| カテゴリ | 変更なし。タスクと独立して併存 |
| メモ | 変更なし。タスクの wiki はメモとは別管理 |
| 中断記録 | 変更なし。中断はポモドーロ記録に紐づき、タスクとは直接関係しない |
| タイマー設定 | 変更なし |
| 既存の PomodoroLog | 既存レコードの taskId は空文字として扱い、後方互換性を維持 |
| `getAllInitData()` | タスクデータは含めない。IndexedDB 同期で別途初期化する（`getAllInitData()` は既存データのみ） |
| 「前回をコピー」 | taskId もコピー対象に含める（同じタスクの連続ポモドーロが多いため） |
| `saveRecord()` | taskId が指定されている場合、該当タスクの `startedAt` が空なら自動設定する処理を追加（セクション 5.3 参照） |

## 7. クライアントサイド データストア

### 7.1 ストレージの使い分け

| データ | ストレージ | 理由 |
|--------|------------|------|
| プロジェクト / 案件 / タスク | **IndexedDB** | 構造的データ、量が多い、非同期クエリ向き |
| メモ | localStorage | 既存実装を維持（LRU eviction 済み） |
| タイマー状態 / UI 設定 | localStorage | 少量の key-value データ |

### 7.2 IndexedDB 設計

DB名: `gas_pomodoro`

#### オブジェクトストア

| ストア名 | keyPath | インデックス | 内容 |
|----------|---------|-------------|------|
| `projects` | `id` | — | メタデータ（content 除く） |
| `cases` | `id` | `projectId` | メタデータ（content 除く） |
| `tasks` | `id` | `projectId`, `caseId`, `status` | メタデータ（content 除く） |
| `contents` | `id` | `lastAccessedAt` | wiki 本文キャッシュ `{ id, content, updatedAt, lastAccessedAt }` |
| `syncMeta` | `key` | — | 同期メタデータ（最終同期時刻など） |

- `contents` ストアはプロジェクト・案件・タスクの wiki 本文を共通で格納する。ID はそれぞれのエンティティ ID をそのまま使用（UUID なので衝突しない）。
- `lastAccessedAt` はノード表示時に更新する（LRU eviction 用）。

### 7.3 同期メカニズム

```
[IndexedDB] ←→ [google.script.run] ←→ [Google Sheets]
   ローカル         サーバー呼び出し          永続ストア
```

#### 基本方針
- **メタデータ（content 除く）**: 初回にサーバーから一括取得し IndexedDB に格納。以降は差分同期。UI は IndexedDB から直接表示する。
- **content（wiki 本文）**: ノード選択時にオンデマンドで取得し IndexedDB にキャッシュ。初回一括取得は行わない（データ量が大きいため）。
- **書き込み**: IndexedDB に即時反映 → バックグラウンドでサーバーへ同期。

#### 同期フロー

1. **ローカル→サーバー（書き込み時）:**
   - IndexedDB のレコードに `_dirty: true` フラグを付与。
   - バックグラウンドで `google.script.run` を呼び出してサーバーに反映。
   - 成功後に `_dirty` フラグを解除。
   - 失敗時はリトライキューに入れる。

2. **サーバー→ローカル（メタデータ同期）:**
   - タブ起動時に `getAllTaskData()` で全メタデータを一括取得し、IndexedDB と比較・更新する。
   - Google Sheets にはインデックスがなく差分クエリでもサーバー側は全行スキャンが必要なため、差分同期ではなく全件取得・比較方式を採用する（実装のシンプルさを優先）。

3. **サーバー→ローカル（content 同期）:**
   - ノード選択時に `contents` ストアを確認。キャッシュがあり `updatedAt` が一致すればそのまま使用。
   - キャッシュがない、または古い場合はサーバーから `getXxxContent(id)` で取得しキャッシュ更新。

4. **競合解決:**
   - ローカルに未同期の変更（`_dirty: true`）がなければ、サーバーを優先。
   - ローカルに未同期の変更がある場合は、メモ機能と同様にユーザーに選択を促す（「ローカルの変更を保持」/「サーバーの内容で上書き」）。

### 7.4 GAS ラグに対する UX 戦略

GAS の `google.script.run` は 1〜5 秒のラグがあるため、以下の対策を行う。

#### Optimistic UI（楽観的更新）

すべての書き込み操作は IndexedDB に即時反映し、UI を即座に更新する。サーバー同期はバックグラウンドで行い、失敗時のみユーザーに通知する。

| 操作 | UI 更新 | サーバー同期 |
|------|---------|-------------|
| タスク追加 | IndexedDB に追加 → ツリー・テーブルに即表示 | バックグラウンド |
| Status 変更 | IndexedDB 更新 → ラベル色を即変更 | バックグラウンド |
| 名前変更 | IndexedDB 更新 → ツリー・テーブルを即反映 | バックグラウンド |
| 並び替え | IndexedDB の sortOrder 即更新 → ツリーを即並び替え | `pendingOrder` パターン |
| content 保存 | IndexedDB に即保存 | debounce 30秒後にサーバーへ |
| アーカイブ | IndexedDB の isActive=false → ツリーから即非表示 | バックグラウンド |

#### 初回ロード

1. IndexedDB にデータがあれば、まず IndexedDB から表示（即時）。
2. バックグラウンドでサーバーから最新メタデータを取得し、差分があれば IndexedDB と UI を更新。
3. 初回（IndexedDB が空）の場合はローディング表示 → サーバーから取得 → 表示。

#### content 遅延ロード（メモタブと同方式）

1. ノード選択時、IndexedDB の `contents` ストアにキャッシュがあれば **readOnly モードで即表示**。
2. キャッシュがない場合は、エディタにローディングインジケーターを表示しつつサーバーから取得。
3. サーバーから `getXxxContent(id)` で最新の content と `updatedAt` を取得し、キャッシュと比較・競合解決。
4. サーバー比較が完了した時点で **editable モードに切り替え**。
5. これにより、ユーザーが古いキャッシュを編集してサーバー側の変更を上書きしてしまう事故を防ぐ。

#### 時間集計

- タスクメタデータに `_cachedTimeSeconds` フィールドをローカル専用（サーバー非送信）で持たせ、テーブルの作業時間列はこの値から表示する。
- `getAllTaskData()` のレスポンスにタスクごとの集計時間を含め、同期時に `_cachedTimeSeconds` を正確な値で上書きする。
- ポモドーロ記録保存時に、該当タスクの `_cachedTimeSeconds` をクライアント側でインクリメンタルに加算する（毎回サーバーに全件集計を要求しない）。

#### 同期中の編集保護

編集は一切ブロックしない（GAS ラグでフリーズさせない）。代わりに以下のルールで整合性を保つ。

**1. 同一レコードの同期キュー（coalesce）:**
- 同じレコードに対する複数の変更は、最新の状態のみをサーバーに送信する。
- 例: Status を ToDo → Doing → Review と素早く変更した場合、サーバーには Review のみ送信。
- 実装: 同期キューにレコード ID をキーとして保持し、同期開始時に IndexedDB から最新の状態を読み取って送信。

**2. 作成→編集の依存順序:**
- 新規作成したレコードを即座に編集した場合、作成の同期が完了するまで編集の同期を保留する。
- 実装: 新規作成レコードに `_pendingCreate: true` フラグを付与。作成同期完了後にフラグを解除し、待機中の編集同期を実行。

**3. content の読み込み時保護:**
- content 表示時は readOnly → サーバー比較完了後に editable に切り替える（メモタブと同方式）。
- editable に切り替わった後の編集中は、バックグラウンドでのサーバー更新チェックを行わない。
- ノード切り替え時に未保存の変更があれば、IndexedDB に保存してから次のノードに移動する。

**4. アーカイブ・削除の即時性:**
- アーカイブはローカルで即時反映し、元に戻す操作（Undo）は提供しない。サーバー同期が失敗した場合のみ、ツリーに再表示してエラー通知する。

**5. リロード・タブクローズ時の保護:**
- `beforeunload` イベントで未同期の content を IndexedDB に flush する（メモの既存実装と同方式）。
- リロード後、IndexedDB に `_dirty` レコードがあれば自動的に同期キューに入れてリトライする。

**6. 作成→アーカイブの依存順序:**
- `_pendingCreate` 状態のレコードをアーカイブした場合、作成同期完了後にアーカイブ同期を実行する（作成→編集と同じ依存順序ルールを適用）。

**7. 親ノードのアーカイブ:**
- プロジェクトをアーカイブする場合、配下の案件・タスクに未同期の変更があれば先にそれらを同期してからプロジェクトのアーカイブを同期する。
- ローカル UI 上は即座に全て非表示にする（Optimistic UI）。

**8. リトライ戦略:**
- リトライは最大5回、指数バックオフ（1秒 → 2秒 → 4秒 → 8秒 → 16秒）。
- 5回失敗後はリトライを停止し、同期エラーとしてユーザーに通知。手動リトライボタンを表示。
- リトライ中に同じレコードへの新しい変更があれば、最新の状態で上書き（coalesce ルール適用）。

**9. IndexedDB 消失への対応:**
- タブ起動時に IndexedDB のデータ存在チェックを行う。空であればサーバーから全メタデータを再取得する（初回ロードと同じフロー）。
- 未同期データが IndexedDB ごと消失した場合はデータロスとなるが、GAS 個人利用の前提上、許容する。サーバー（Google Sheets）が正のデータソースであり、ローカルはキャッシュとして扱う。

**10. startedAt 自動設定と `_pendingCreate`:**
- ポモドーロ保存時に紐づけたタスクが `_pendingCreate` 状態の場合、ポモドーロの `saveRecord()` は通常通りサーバーに保存するが、startedAt 自動設定はスキップされる（サーバーにタスク行がないため）。
- タスクの作成同期完了後、クライアント側で startedAt が空のタスクに対してポモドーロの startTime をセットし、`updateTask` で同期する。

#### 同期状態の表示

- 未同期の変更がある場合、タスクタブのヘッダーに同期インジケーター（例: 薄い回転アイコン）を表示。
- 同期エラー時は赤いインジケーターで通知し、リトライボタンを提供。

### 7.5 IndexedDB 解放戦略

#### A. content の LRU eviction

- `contents` ストアの `lastAccessedAt` インデックスを使い、最終アクセスが古い順に削除する。
- **発動条件**: `contents` ストアのレコード数が上限（例: 100件）を超えた場合、または `contents` への書き込み時に `QuotaExceededError` が発生した場合。
- **削除対象**: `_dirty` フラグがないレコード（未同期の変更がないもの）のうち、`lastAccessedAt` が最も古いものから削除。
- **削除しないもの**: 現在エディタに表示中の content、未同期の content。
- 削除された content はノード選択時にサーバーから再取得される（content 遅延ロードの通常フロー）。
- メタデータ（projects/cases/tasks ストア）は LRU 対象外（サイズが小さいため常に保持）。

#### B. アーカイブ済みデータの自動パージ

- メタデータ同期時に、サーバーから取得したデータのうち `isActive=false` のレコードは IndexedDB に保存しない。
- 既に IndexedDB にあるレコードが `isActive=false` に変わった場合、同期完了後に IndexedDB から削除する。
- 対応する `contents` ストアのキャッシュも合わせて削除する。
- アーカイブ済みデータの表示が必要な場合は、その都度サーバーから取得する。

## 8. 制約・留意事項

- **セル上限**: Google Sheets の1セル50,000文字制限が wiki コンテンツに適用される。
- **GAS 実行時間**: サーバー呼び出しは6分制限内で完了する必要がある。集計クエリが大量データの場合はキャッシュを活用する。
- **IndexedDB**: ブラウザの private browsing モードでは容量制限がある場合がある。基本的にサーバー（Google Sheets）が正とし、IndexedDB はキャッシュ兼オフラインストアとして扱う。
